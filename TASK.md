# Тестовое задание. Backend, Python

Необходимо реализовать асинхронное веб приложение в парадигме REST API.

Время выполнения задачи: **5 дней**.

---

## Стек

- **База данных:** PostgreSQL
- **ORM:** SQLAlchemy
- **Веб-фреймворк:** Sanic (рекомендуемый, допускается альтернативный НЕ Django)
- **Контейнеризация:** Docker Compose

---

## Сущности

- **Пользователь** — базовая учётная запись
- **Администратор** — пользователь с правами управления
- **Счет** — имеет баланс, привязан к пользователю
- **Платеж (пополнение баланса)** — хранит уникальный идентификатор и сумму пополнения счета пользователя

---

## Функциональные требования

### Пользователь

- Авторизоваться по email/password
- Получить данные о себе (id, email, full_name)
- Получить список своих счетов и балансов
- Получить список своих платежей

### Администратор

- Авторизоваться по email/password
- Получить данные о себе (id, email, full_name)
- Создать / удалить / обновить пользователя
- Получить список пользователей и список его счетов с балансами

### Webhook (обработка платежей)

Должен быть реализован роут, эмулирующий обработку вебхука от сторонней платежной системы.

**Структура JSON-объекта:**

| Поле | Описание |
|------|----------|
| `transaction_id` | Уникальный идентификатор транзакции в «сторонней системе» |
| `account_id` | Уникальный идентификатор счета пользователя |
| `user_id` | Уникальный идентификатор пользователя |
| `amount` | Сумма пополнения счета пользователя |
| `signature` | Подпись объекта |

**Формирование подписи:**

`signature` — SHA256 хеш строки, состоящей из конкатенации значений объекта **в алфавитном порядке ключей** и «секретного ключа» из конфигурации проекта:

```
{account_id}{amount}{transaction_id}{user_id}{secret_key}
```

Пример для `secret_key = gfdmhghif38yrf9ew0jkf32`:

```json
{
  "transaction_id": "5eae174f-7cd0-472c-bd36-35660f00132b",
  "user_id": 1,
  "account_id": 1,
  "amount": 100,
  "signature": "7b47e41efe564a062029da3367bde8844bea0fb049f894687cee5d57f2858bc8"
}
```

**Логика обработки вебхука:**

1. Проверить подпись объекта
2. Проверить, существует ли у пользователя указанный счет — если нет, создать
3. Сохранить транзакцию в базе данных
4. Начислить сумму транзакции на счет пользователя

Транзакции являются уникальными: начисление суммы с одним `transaction_id` должно производиться только один раз.

## Миграция и тестовые данные

В миграции должны быть созданы:

- Тестовый пользователь
- Счет тестового пользователя
- Тестовый администратор

---

## Развертывание

Необходимо реализовать Docker Compose конфигурацию, состоящую из:

- Сервис PostgreSQL
- Сервис приложения

К заданию должна прилагаться краткая инструкция по запуску в двух вариантах:

1. С использованием Docker Compose
2. Без Docker Compose (локально)

В инструкции должны быть указаны email/password для предустановленных пользователя и администратора.
